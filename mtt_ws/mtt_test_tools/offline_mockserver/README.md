# MTT CAN Bus Mock Server

This project provides a set of tools to simulate the CAN bus of an MTT all-terrain vehicle. It is designed for driver software development and testing without requiring physical hardware.

The server simulates the vehicle's basic logic, responds to control commands, and sends status data (BMS, encoder) in accordance with the MTT CAN bus specification.

## Features

- **Vehicle Behavior Simulation**: Simulates basic acceleration and braking logic, prioritizing brake.
- **Control Frame Listening**: Accepts commands on CAN IDs 0x100 (external) and 0x101 (joystick).
- **Status Data Broadcasting**: Periodically sends vehicle status CAN frames:
  - **Main Data (0x2FF)**: Instantaneous speed (tachometer), cumulative distance, and temperature.
  - **BMS Data (0x600-0x602)**: Cell temperatures, state of charge (SOC), voltage, and current.
- **Helper Scripts**: Provides scripts for compilation, test launching, and bus monitoring.

## Prerequisites

Make sure the following tools are installed on your Linux system:

- `g++` (C++ compiler)
- `cmake`
- `can-utils` (for candump and cansend)

You can install them on a Debian/Ubuntu system with:

```bash
sudo apt-get update
sudo apt-get install build-essential cmake can-utils
```

## Compilation

To compile the simulation server, simply run the build script:

```bash
./build_mtt_mock.sh
```

This will create a `build/` directory containing the `mtt_mock_server` executable.

## Usage

### 1. Virtual CAN Interface Configuration

For testing, it is highly recommended to use a virtual CAN interface (vcan). The test script can do this for you, but if you want to do it manually:

```bash
sudo modprobe vcan
sudo ip link add dev vcan0 type vcan
sudo ip link set up vcan0
```

### 2. Launch the Simulation Server

The easiest way to start is to use the test script. It will ensure that the vcan0 interface is active and launch the server:

```bash
./test_mtt_system.sh
```

The server will start listening for commands and sending status data.

### 3. Send Test Commands

Open another terminal and use `cansend` to send a control frame.

**Example: Move forward with accelerator at 50%**

- **CAN ID**: 100 (or 101)
- **Payload**: `00 80 80 7F 00 80 00 00`
  - Byte 1 (0x80): Safety unlocked (Bit 7 = 1) and Headlight on (Bit 6 = 0)
  - Byte 2 (0x80): Accelerator at ~50% (128 out of 230)

```bash
cansend vcan0 100#0080807F00800000
```

### 4. Monitor CAN Bus

Open a third terminal to see all CAN traffic generated by the server and your commands:

```bash
./monitor_can.sh
```

You will see frames 0x2FF, 0x600, 0x602 sent by the server, as well as frames 0x100/0x101 that you send.

### 5. Additional Test Commands

**Test reverse direction with throttle:**
```bash
cansend vcan0 100#00A0807F00800000
```

**Test braking:**
```bash
cansend vcan0 100#0080007F50800000
```

**Test ID 0x101 compatibility:**
```bash
cansend vcan0 101#0080FF7F00800000
```

**Test safety locked (no movement):**
```bash
cansend vcan0 100#0000807F00800000
```

## CAN Frame Decoding

### Outgoing Frames (Sent by Mock Server)

**0x2FF - Main Data Frame:**
- Bytes 0-1: Temperature A, Temperature B (°C)
- Bytes 2-3: Instant tachometer (16-bit signed, big-endian)
- Bytes 4-7: Cumulative ticks (32-bit unsigned, big-endian)

**0x600 - BMS Frame 1 (Battery Cell Temperatures):**
- Bytes 0-7: 4 cell temperatures (16-bit signed each, big-endian)

**0x602 - BMS Frame 3 (Battery Status):**
- Byte 0: State of Charge (SOC) percentage
- Bytes 1-2: Current (16-bit signed, big-endian, scale: 0.01A)
- Bytes 3-4: Voltage (16-bit unsigned, big-endian, scale: 0.01V)

### Incoming Frames (Control Commands)

**0x100/0x101 - Control Frame:**
- Byte 0: Vehicle Type
- Byte 1: Global switches (Bit 7: Safety, Bit 6: Headlight, Bit 5: Direction)
- Byte 2: Throttle (0-255)
- Byte 3: Winch (0x7F = neutral)
- Byte 4: Brake (0-255)
- Byte 5: Steering (128 = center)
- Byte 6: Direction/Mode switches (Bit 0: Closed-loop steering)
